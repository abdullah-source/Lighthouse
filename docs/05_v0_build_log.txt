Also landed: the "fast loop / slow loop" framing for differentiation
(answers the prof's question about how intentional content changes
affect new model releases):

  - Fast loop = retrieval-grounded models (Perplexity, ChatGPT
    Search, Claude Search, Gemini Grounding). Content changes
    propagate in hours-days.
  - Slow loop = training-grounded base models. Content propagates
    in 3-12 months through training cycles, authoritative
    citations, schema, Common Crawl weight.
  - Most competitors only optimize the fast loop. We do both.

2. v0 ARCHITECTURE
================================================================

Five stages glued by a SQLite database. Pure CLI script, no web.

  CLI args (--brand, --category)
    -> config.py: load .env, validate keys
    -> db.py: init SQLite schema
    -> queries.py: 1 Sonnet call generates 20 buyer queries
    -> probe.py: async fan-out, each query x 2 models = 40 calls
    -> parse.py: async fan-out, Haiku w/ JSON tool-use per response
    -> report.py: aggregate, print terminal report

Models locked in config.py:
  - Query generation:    claude-sonnet-4-6
  - Probing (Anthropic): claude-sonnet-4-6
  - Probing (OpenAI):    gpt-5
  - Parsing:             claude-haiku-4-5-20251001

Stack:
  - Python 3.13
  - anthropic, openai, httpx, python-dotenv
  - SQLite (stdlib, no ORM)
  - asyncio for parallel API calls
  - argparse for CLI

6. DEBUGGING WE HIT
================================================================

ISSUE A — OpenAI key not loading
  Symptom: Anthropic loaded YES, OpenAI loaded NO after editing .env
  Root cause: line in .env still started with "#", so it was a
              comment and python-dotenv ignored it.
  Fix:   removed the leading "#" from the OPENAI_API_KEY= line.

ISSUE B — OpenAI API call rejected with BadRequestError
  Symptom: "Unsupported parameter: 'max_tokens' is not supported
            with this model. Use 'max_completion_tokens' instead."
  Root cause: GPT-5 is a "reasoning-class" model and uses the new
              parameter name. Legacy GPT-4 code patterns don't work.
  Fix:   renamed `max_tokens` -> `max_completion_tokens` in
         probe.py _probe_openai().

ISSUE C — OpenAI call succeeds but returns empty string
  Symptom: finish_reason='length', completion_tokens=800,
           reasoning_tokens=800, accepted_prediction_tokens=0.
  Root cause: GPT-5 spends tokens on internal chain-of-thought
              BEFORE producing visible output. With max_completion_tokens
              = 800, all budget went to reasoning, leaving zero for
              the actual answer.
  Fix:   - bumped max_completion_tokens to 2000
         - added reasoning_effort="low" to cap internal reasoning
         Confirmed working: 320 reasoning + 250 output tokens,
         clean response with 3 specific shoe brands recommended.

ISSUE D (cosmetic, not user-facing) — main.py print message
  Said "queries x 2 models" even when running Claude-only.
  Fixed to just "Probing N queries..." regardless of mode.

ISSUE E (IDE-only, not runtime) — VSCode red squiggles on imports
  Symptom: "Cannot find module 'anthropic'" / "Cannot find module 'openai'"
  Root cause: VSCode's Python language server was using system Python
              (~/Library/Python/3.13/) instead of the venv where the
              packages are installed.
  Fix:   created .vscode/settings.json pointing at .venv/bin/python.
         Reload window after editing settings.json.
